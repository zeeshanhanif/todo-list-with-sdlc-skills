# design.md Guide (the Agent-Ready Design System)

`docs/design.md` is the **render-time** document: what a coding agent loads when
building UI. Its reader implements screens; every line should either give a
concrete value or a rule that keeps generated UI on-system. Reference it from
CLAUDE.md so every UI-building session inherits it.

Authority split: design.md (with tokens.json) is the **single source of truth**
for anything design-system. ux-foundations.md references it and never restates
token values — that's how the two files are kept from drifting.

Style rules throughout: concrete values always (hex, px, weights — never
"generous spacing"); semantic role names with raw values behind them; every
interactive component specified per state; honest about what's undefined.

Use this structure:

```markdown
# Design System: <Product Name>

> Source of truth for all UI implementation, with tokens.json.
> Generated by ux-foundations · Mode: <research | images | design file | tool>
> Last updated: <date>

## 1. Visual Theme & Atmosphere
A short paragraph an agent can *feel*: the personality (two or three
adjectives), density, how the product earns its audience's trust. Sets the
judgment calls the rest of the file can't enumerate.

## 2. Design Tokens
Semantic, role-based tokens. Raw values live behind roles so theming (dark
mode) works. **Derived from tokens.json — same values, embedded here as CSS
custom properties for direct use:**

### Colors
| Role | Token | Value | Usage |
| :--- | :---- | :---- | :---- |
| Primary | `--color-primary` | #146128 | Primary brand surfaces, nav |
| Primary hover | `--color-primary-hover` | #22963F | Hover/active on primary |
| CTA | `--color-cta` | #E8A020 | High-priority actions only |
| Surface | `--color-surface` | #FFFFFF | Cards, panels |
| Text | `--color-text` | #111111 | Headings, body |
| … success / warning / danger / border / muted … |

```css
:root {
  --color-primary: #146128;
  --color-cta: #E8A020;
  /* …full set, generated from tokens.json… */
}
```
Contrast note per pairing that matters (e.g., "CTA on Surface: 4.6:1 — passes
AA"). Dark-mode set if in scope, as the same roles re-valued.

### Typography
Families (with fallback stacks), the scale as a table (step, size, line-height,
weight, usage), and rules ("Outfit for all UI text; mono only for code").

### Spacing, Radius, Elevation
The scales as explicit value lists, each with usage notes ("section gap: 48px;
component gap: 24px; intra-component padding: 16–24px").

## 3. Layout & Grid
Container widths, grid, breakpoints (with what collapses where), page-shell
structure (header/nav/content/footer heights and behavior), touch-target
minimums.

## 4. Components
Per component: variants and **all states** (default / hover / focus / active /
disabled / loading / error where applicable), with concrete values. Buildable
specs, not one-liners:

**`button-primary`** — `--color-cta` fill, white text, radius 8px, padding
14×24px, height 48px, weight 500. Hover: `--color-cta-hover`. Focus: 2px
visible ring `--color-focus`. Disabled: 40% opacity, cursor not-allowed.

Cover the core set (actions, forms + validation display, feedback, structure,
navigation primitives) and the empty/loading/error conventions for data views.

### Per-surface variants
Where a surface overrides (denser table row height on the admin portal, larger
touch targets on mobile) — as overrides of the named tokens/components above,
never a parallel system.

## 5. Accessibility Rules  ← non-negotiable, from SRS NFRs (cite IDs)
The standard (e.g., WCAG 2.2 AA per NFR-USE-001) made mechanical: minimum
contrast per token pairing, focus always visible (spec the ring), touch targets
≥ 44px, full keyboard operability, reduced-motion respect, semantic structure.

## 6. Voice & Microcopy
Two or three do/don't pairs for UI text; error-message tone; per-surface tone
notes if they differ.

## 7. Do's and Don'ts
The compressed rule list that keeps agent output on-system:
- **Do** use tokens for every color/space/radius — never raw values.
- **Do** pick spacing only from the scale.
- **Don't** invent component variants; extend via the defined axes.
- **Don't** use colors outside the palette.
(Extend with the system's own specific rules.)

## 8. Agent Quick Reference
A cheat-sheet an agent can follow without re-reading the file: the 8–12 most-
used token/value pairs, the default component choices ("primary action →
button-primary"), and the one-line layout recipe.

## 9. Provenance & Known Gaps
- **Provenance:** which source mode produced this system, from what, and its
  fidelity (tool-pulled values are exact; image extraction is approximate;
  research is designed-from-requirements). Record any SRS-driven adjustments
  ("ingested palette's CTA failed AA; darkened from #F0B84D to #E8A020").
- **Known Gaps:** what the system doesn't yet define (states not extractable,
  breakpoints assumed, motion undefined) — honest, so a later session knows
  what's decided versus defaulted.
```

## Right-sizing

Single-surface internal tool: compress sections 1/6 to a few lines, keep 2–5, 7–9
tight. Multi-surface product: full treatment, with per-surface variants (§4)
carrying real weight. In every case: **complete tokens, complete states,
explicit gaps** — those three are never compressed.
